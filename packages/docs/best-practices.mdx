Follow standardized practices to ensure your documentation remains maintainable,
scalable, and easy for users to navigate.

This guide synthesizes principles from game design, modern documentation
frameworks, and technical writing standards.

---

## Philosophy of documentation

Great documentation enforces comprehension, anticipates user needs, and
validates their environment.

### Frictionless onboarding

Prioritize end-user value over underlying technology. Get the user a result
within 60 seconds. Provide single-command initializers (for example,
`npx @trustgraph/config`) to shift the user’s experience from environment
configuration to product usage.

Create a mandatory, zero-stakes initial sequence to train the user on the UI
logic and core interactions before exposing complex material.

### Productive friction

Frictionless design causes cognitive stagnation. Introduce desirable difficulty
to prevent passive reading.

- Use heuristic stress tests. Present counter-hypotheses or use Socratic AI
  agents to force users to justify their approach.
- Embed unresolved vulnerabilities directly into theory modules to challenge
  assumptions.
- Reward long-term systemic investment by dynamically unlocking advanced
  documentation based on the user's historical interaction and peer review.

### Zero-player tutorials

Transfer knowledge diegetically through system behavior rather than explicit
textual manuals.

- Present fully operational models where the system executes highly optimized,
  synergistic code in real-time. Expect the user to observe, reverse-engineer,
  and mimic optimal execution.
- Combat cognitive overload with progressive disclosure. Hide global elements
  and reveal only the specific tools and libraries needed for the immediate
  task.

### Conceptual anchoring and validation

Explain the problem a feature solves before explaining how the feature works.
Use comparison tables to contrast the old way with the new way.

Include real-time visual feedback loops (like a 3D workbench) that validate the
user's API calls. Complexity, especially in graph theory and heavy tech stacks,
requires visual validation.

Identify common environmental friction points (like ports, Docker, or storage).
Use warning and note callouts proactively to troubleshoot these issues.

## Information architecture

### The Diátaxis framework

Separate your documentation logically based on the user's intent. Do not mix
concepts with procedures.

| Quadrant          | Focus                  | Goal                                                    | Writing style              |
| ----------------- | ---------------------- | ------------------------------------------------------- | -------------------------- |
| **Tutorials**     | Learning-oriented      | Help the user get started through a guided lesson.      | Educational, step-by-step. |
| **How-to guides** | Task-oriented          | Solve a specific, real-world problem or achieve a goal. | Practical, goal-focused.   |
| **Reference**     | Information-oriented   | Provide technical descriptions of the machinery.        | Neutral, fact-based.       |
| **Explanation**   | Understanding-oriented | Explain the "why" and provide background context.       | Theoretical, discursive.   |

### Hierarchy and layout

Choose a primary navigation pattern in your configuration file that fits your
audience, and keep the hierarchy as shallow as possible.

- **Groups**: Use for a single audience.
- **Tabs**: Separate distinct sections for different audiences.
- **Omnichannel tabs**: Hide platform-specific content (for example, hide web
  setups when viewing a Telegram bot guide).
- **Modular architecture**: Write documentation that mirrors the software
  architecture. Let users swap out database or authentication integration
  instructions without losing the tutorial's overall thread.

## Writing standards

### Voice, tone, and perspective

Write exclusively in the second-person voice ("you").

Use active voice. Active voice is shorter, direct, and avoids obfuscation.
Follow the formula: Actor + Verb + Target.

- _Active:_ The tool parsed the flags.
- _Passive:_ The flags were parsed by the tool.

Use imperative verbs for commands. Commands like "Open the file" are active,
with an implied actor of "you."

Maintain a professional, authoritative, and direct tone.

### Game design document principles

- Avoid impenetrable walls of text. Visualize flowcharts and logic paths.
- Convert comparative or qualitative data into tables.
- Explicitly state what users require to succeed and what is out of scope.
- Follow the rule of one sentence per thought.

### What to avoid

- **Marketing fluff**: Never use words like "powerful," "seamless," "robust," or
  "cutting-edge." Keep documentation factual.
- **Filler phrases**: Eliminate bloat like "it is important to note" or "in
  order to."
- **Editorializing**: Remove subjective modifiers like "obviously," "simply,"
  "just," or "easily."
- **AI-typical patterns**: Avoid generic introductions, stilted phrasing, and
  concluding summaries that restate previous sections.

### Formatting and code

Use sentence case for headings and code block titles ("Getting started", not
"Getting Started"). Avoid numbered headings unless conveying a strict sequence.

Provide one exceptionally clear code example rather than multiple slight
variations.

Use realistic placeholder values. Never use "foo" or "bar".

Specify the language identifier tag for syntax highlighting.

```bash
# Example syntax highlighting
npx create-react-app my-app
```

Test that the code works before publishing.

Use pre-filled `.env` examples. Provide direct links to relevant dashboards when
users require API keys.

## Components and organization

### Content components

Structure your content using UI components to manage cognitive load.

| Need                            | Component                      |
| ------------------------------- | ------------------------------ |
| Hide deep-dive details          | **Accordions / Collapsibles**  |
| Display massive code payloads   | **Expandable blocks**          |
| Let user choose a language      | **Tabs**                       |
| Walk through instructions       | **Steps**                      |
| Show code in multiple languages | **Code groups**                |
| Present API parameters          | **ParamField / ResponseField** |

Use callouts by severity:

- **Note:** Supplementary info; safe to skip.
- **Info:** Contextual info.
- **Tip:** Best practices.
- **Warning:** Destructive actions or gotchas.
- **Success/Check:** Confirmation state.

### Reusable content (snippets)

Use snippets for exact content that appears on multiple pages, or to manage
complex shared components.

Do not use snippets for slight variations per page. This creates unsustainable
properties.

### File and frontmatter conventions

Always include a `title` in the frontmatter. Provide a `description` for SEO.

Use exactly `kebab-case.mdx` for file names.

Use root-relative paths without extensions internally (for example,
`/getting-started/quickstart`). Use full absolute URLs for external links. Keep
media organized in a dedicated `/images` directory.

## Document APIs

Choose your API documentation strategy based on your team's context:

1. **OpenAPI specification**: Auto-generate endpoints from your spec. This is
   the most efficient and sustainable approach.
2. **Manual pages**: Write manual endpoints when you lack a spec but require
   full control.
3. **Hybrid**: Auto-generate standard endpoints, and write manual guides for
   complex workflows.

## Documentation workflow

Before creating documentation, follow these steps:

1. **Understand task and project**: Review your core platform configuration
   (`docs.json`). Know the navigation setup, theme, and routing constraints.
2. **Check for existing content**: Always search first. Update an existing page
   instead of adding a new one.
3. **Read surrounding content**: Read two to three existing pages to internalize
   the established voice, tone, and level of detail.
4. **Plan**: Outline the necessary updates. Lead with what the reader
   accomplishes.
5. **Write**: Put the most important information first. Keep paragraphs
   scannable. Format with active voice. Use `TODO` comments for uncertain
   details.
6. **Update navigation**: Bind new content paths to the site's structural
   configuration file.
7. **Verify**: Check frontmatter, language tags, broken internal links, and tone
   rules.

## Communications and social strategy

Every technical update must provide immediate, tangible value while reinforcing
your brand's authority. Translate technical signal into high-fidelity narrative.

- **Signal over noise**: Skip marketing jargon. Identify the immediate value
  this update gives the user.
- **Reciprocity**: Offer specific value (blueprints, code snippets,
  architectures) before asking for engagement.
- **Mimetic desire**: Highlight adoption by high-impact engineers or ecosystem
  partners.
- **The pratfall effect**: Be radically transparent. Acknowledge past flaws to
  build trust.
- **Open-source transparency**: Be fully transparent about friction (Docker,
  databases, authentication) when documenting self-hosted paths. Users require
  accurate information to choose between local or managed cloud options.
- **Mere exposure effect**: Be consistent in your visual styling and voice.
- **Curse of knowledge**: Remember that what is obvious to the engineer is never
  obvious to the new user.
