The Worlds Platform utilizes a managed neuro-symbolic infrastructure designed
for edge-distributed, agentic memory. It separates a **Control Plane**, or
Console, for management from a **Data Plane**, or Worlds API, for
high-performance execution.

## Control vs. data plane

The platform splits operations into two primary layers:

### 1. Control plane

The **Management Console** acts as the system's brain. It manages identity
through WorkOS, handles organization-level provisioning, and orchestrates World
Server instances.

### 2. Data plane

The **Worlds API Server** handles RDF graph management, SPARQL execution, and
hybrid search. This is the data plane where your information lives.

## Resource hierarchy

Understanding the relationship between **Organizations** and **Worlds** is key
to managing isolation and scaling.

```mermaid
graph LR
    Org[Organization] --> Server[API Server]
    Org --> MetaDB[(Primary DB)]
    Server --> W1[World 1]
    Server --> W2[World 2]
    W1 --> W1DB[(World 1 DB)]
    W2 --> W2DB[(World 2 DB)]
```

### Organizations

The Console manages Organizations as top-level groupings. Provisioning an
Organization initializes a dedicated API server instance, a primary metadata
database, and an admin security boundary.

### Worlds

The Organization's server manages each World as a specific context or graph.

- **Dedicated Storage**: Each World maintains its own secondary libSQL database
  for triples and embeddings.
- **Isolation**: Worlds are sub-resources accessed via the Org's base URL, such
  as `/v1/worlds/{id}`, ensuring zero cross-contamination.

### Repository layout

The server follows a modular layout organized by service and resource:

```text
The Worlds API Server follows a modular layout organized by service and
resource, separating core reasoning logic from the persistence and transport
layers.

Key directories include:

- `lib/`: Shared logic for RDF/SPARQL handling, database management, and
  embeddings.
- `middleware/`: Authentication and rate-limiting logic.
- `models/`: API models and shared types.
- `routes/`: Implementation of the v1 API endpoints.
```

- **Entrypoints**: The server and CLI provide dedicated entrypoints for
  initialization and CLI command execution.
- **Import alias**: The `#/` alias maps to `./` within each package.

## Design principles

## Polymorphic resource managers

A key design feature is the use of hot-swappable resource managers. The core
logic remains identical, while the implementation swaps based on the
environment:

| Resource     | Local dev implementation | Production implementation |
| :----------- | :----------------------- | :------------------------ |
| **Compute**  | Local child processes    | Managed edge runtime      |
| **Storage**  | Local SQLite files       | Remote libSQL             |
| **Identity** | Mock identity file       | Managed Identity Service  |

This pattern allows the entire stack to run locally with zero cloud
dependencies.
