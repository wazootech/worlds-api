# Database layer

The Worlds platform utilizes a dual-scope database architecture to separate
management logic from world-specific data.

## Two database scopes

The system manages two distinct types of database instances:

- **Main database** (single instance):
  - **Tables**: `organizations`, `worlds`, `service_accounts`, `invites`,
    `rate_limits`, `usage`.
  - **Initialization**: Created by `initializeDatabase(client)` in
    `lib/database/init.ts`.
- **World databases** (one per world):
  - **Tables**: `chunks` (with FTS and vector index), `triples`, `logs`.
  - **Initialization**: Created by `initializeDatabase(client)` in
    `lib/database/init.ts` upon world creation.

## DatabaseManager interface

Defined in `packages/server/lib/database/manager.ts`, the `DatabaseManager`
interface provides a common API for world database lifecycle management:

- `create(id: string): Promise<ManagedDatabase>`: Creates a new world database
  and initializes its schema.
- `get(id: string): Promise<ManagedDatabase>`: Retrieves a `ManagedDatabase`
  instance for the specified world.
- `delete(id: string): Promise<void>`: Removes the database resources for the
  specified world.

### Implementations

- **FileDatabaseManager**: Stores each world as `{id}.db` in the
  `WORLDS_BASE_DIR` (defaults to `./worlds`).
- **TursoDatabaseManager**: Interfaces with the Turso API to manage remote
  databases and tokens.
- **MemoryDatabaseManager**: Utilizes in-memory clients for testing purposes.

## SQL and code generation

The platform uses a structured approach to SQL management and type safety:

- **Queries**: Live in `.sql` files near their respective resources.
- **Code generation**: The `sql-embedder` utility generates `queries.sql.ts`
  files. Run `deno task generate` to update these files.
- **DDL execution**: `core/init.ts` imports table and index names from generated
  files to ensure correct execution order.
